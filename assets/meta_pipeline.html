The Meta-Loop: a general model of a self-tuning process

This is not a description of a pipeline. A pipeline is just one revolution of the cycle; the real object is the improvement loop. A product = a set of independent self-tuning loops; the LLM inside them is secondary and replaceable, the architecture remains.

Genesis of a loop: where the roles come from

Every process starts as human work → a temporary reviewer appears → accumulated decisions turn into a deterministic process → the reviewer is no longer needed → the director stops intervening in operations. A loop is mature when its recurring decisions are fully expressed in code and rules. Every role is technical debt: it exists only while the process is not yet mature enough.
  • The law of decision gravity: every decision tends to sink one level down — director → hypothesis → rule → code → conveyor. A month later the director has forgotten that category of decisions exists.
  • The main flow is not actions but knowledge: the loop turns complaints and verdicts into rules, rules into code.
  • The director is not temporary. He stops running operations — but he alone changes the goals.
  • The manager's objective function (minimize simultaneously): share of decisions made via LLM · complaints · doubts · processing cost · decision latency.
  • Complaints outweigh any internal metric — an external source of truth; without it the loop optimizes its own grades.
▼ roles of one revolution

1 · CONVEYOR — the work process determinism + local models

  • Processes the entire incoming stream (all domain sources — no exceptions)
  • Applies confident decisions itself, immediately and reversibly; passes uncertain ones on
  • Runs on a schedule, lives locally, costs little
Contract: every decision → journal (what, why, which signals); every action reversible; caps on volume per run.
▼ the uncertain zone

2 · WORKER-REVIEWER — temporary controller LLM (being phased out)

  • Judges the uncertain zone: apply / defer to doubts / discard — verdicts applied immediately, reversibly
  • Reviews the conveyor as a whole: also spot-rechecks the confident zone (sampled audit), watches run metrics
  • Audit disagreeing with the conveyor → a flag in metrics (not a unilateral override): the optimization manager and director decide
Contract: verdict + reason → journal; audit disagreements → run metrics; its own quality is measured the same way; exists until the conveyor learns to do it itself.
▼ applied actions + remaining doubts

3 · STOREFRONT + COMPLAINTS director (owner)

  • The result works immediately (goods on the shelves); the director need not approve anything in advance
  • Complaint: one undo action + a free-text "why this was wrong" — the highest ground truth
  • Observation panel: live quality by tier, an action feed with reasons, degradation alerts, the doubt queue
Contract: silence = consent; complaints outweigh any internal metric; everything visible in one place in 5 minutes.
▼ journals + metrics + complaints ▼

4 · OPTIMIZATION MANAGER — analysis and decisions strong LLM

  • Breakdown: assess the conveyor's work in numbers; process problems and worker-reviewer problems — kept separate; trend against past reviews
  • Decisions: process changes (priority to deterministic ones that displace the worker); each with an expected effect and a way to verify it
  • Escalations to the director: product forks, trade-offs, dead-ends — not decided alone
Contract: honors the director's directives; does not repeat its own past hypotheses; decisions → cards in a shared backlog; periodic + on-demand.
▼ auto-chain: decisions go straight to building

5 · BUILDER — implementation strong LLM, agentic

  • Takes the top decision → implements it with a minimal diff → mandatory eval gate (metrics no worse than baseline)
  • Gate passed → the change goes live, the next conveyor run works the new way; failure → automatic revert and an honest "why"
Contract: edits only its own process files (whitelist); backup before editing; result goes to the director for acceptance (only he marks Done); one pass — one decision.
⟳ loop closed: the conveyor got better → less work for the worker → fewer complaints for the director

👔 Director: two controls

  • Complaints — pointed corrections of facts ("this doesn't go there, because…")
  • Directives — the vector for the optimization manager (priorities, prohibitions, goals); edited in one line, re-read every cycle
  • Acceptance: Review → Done; escalations require his answer

📏 Loop maturity criteria

  • Share of decisions without a cloud LLM → rising
  • Live precision of tiers ≥ 95–98% → the worker is removed from the link
  • Complaints → toward zero; director involvement ≈ 5 min/day
  • Removing control tiers — by numbers only, bottom-up
